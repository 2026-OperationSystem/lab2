# xv6 시스템 콜 실행 흐름 분석 (exit / exit2 기준)

본 문서에서는 사용자 프로그램(User Program)이 시스템 콜을 호출했을 때 커널(Kernel) 모드로 진입하여 실제 로직이 수행되기까지의 전체 흐름을 xv6의 구조적 관점에서 분석합니다. `exit()` (및 새로 구현한 `exit2()`) 시스템 콜의 실행 과정을 예시로 설명합니다.

## 시스템 콜 실행 단계별 분석

### 1. `user.h` (사용자 인터페이스 선언)
- **역할**: 유저 레벨(User Level) 프로그램이 호출할 수 있는 시스템 콜의 함수 원형(Prototype)을 선언합니다.
- **분석**: 응용 프로그램(예: `testwait.c`, `team1id.c`)은 `#include "user.h"`를 통해 `exit()` 또는 `exit2(int status)` 함수를 사용할 수 있게 됩니다. 이는 커널로 진입하기 위한 사용자 공간의 창구 역할을 합니다.

### 2. `usys.S` (시스템 콜 인터럽트 발생)
- **역할**: 어셈블리 매크로를 통해 실제 시스템 콜 번호를 레지스터(`eax`)에 저장하고, 소프트웨어 인터럽트(`int $T_SYSCALL`, xv6에서는 `int 64`)를 발생시켜 커널 모드로 진입(트랩)합니다.
- **분석**: `user.h`에 선언된 함수가 호출되면 실제로는 컴파일 시 링크된 `usys.S` 내의 매크로 코드가 실행됩니다. `SYSCALL(exit2)` 매크로는 `eax` 레지스터에 `SYS_exit2` 번호를 적재하고 `int 64` 트랩 명령어를 실행하여 제어권을 하드웨어에서 커널의 트랩 핸들러로 전환합니다.

### 3. `syscall.h` (시스템 콜 번호 매핑)
- **역할**: 각 시스템 콜에 고유한 상수 번호를 매핑하여 식별자로 정의합니다.
- **분석**: `#define SYS_exit 2` 또는 새롭게 추가된 `#define SYS_exit2 22`와 같이 고유 번호가 부여되어 있습니다. `usys.S`에서 레지스터에 넣는 값과 커널 내부에서 어떤 시스템 콜이 호출되었는지 식별할 때 이 번호를 공통 규약으로 사용합니다.

### 4. `syscall.c` (시스템 콜 디스패처)
- **역할**: 인터럽트를 통해 커널 모드로 넘어온 후, `eax` 레지스터에 저장된 번호를 확인하여 해당 번호에 매핑된 커널 래퍼(Wrapper) 함수로 분기(Dispatch)합니다.
- **분석**: `syscall()` 함수가 실행되면, 프로세스의 트랩 프레임(Trap frame)에 저장된 `eax` 값을 확인합니다. 이 값을 인덱스로 삼아 함수 포인터 배열인 `syscalls[]`에서 대응되는 함수인 `sys_exit` 또는 `sys_exit2`를 찾아내어 호출합니다.

### 5. `sysproc.c` (유저 인자 처리 및 래퍼 함수)
- **역할**: 커널 영역에서 동작하며, 유저 공간에서 레지스터나 스택을 통해 전달한 인자(Arguments)를 안전하게 커널 공간으로 가져오고 검증한 후, 실제 핵심 로직을 담당하는 내부 함수를 호출합니다.
- **분석**: 기존의 `sys_exit(void)`는 넘겨받을 인자가 없으므로 바로 `exit()`를 호출했습니다. 하지만 새롭게 구현한 `sys_exit2(void)`는 `argint(0, &pid)` (코드상 변수명 pid, 실제로는 status) 함수를 사용하여 유저 스택에서 정수형 인자 값인 종료 상태 코드를 추출하고 예외를 검증한 뒤, 이 값을 실제 운영체제 로직을 수행하는 `exit2()`에 전달합니다.

### 6. `proc.c` (실제 프로세스 관리 로직 수행)
- **역할**: 운영체제의 핵심적인 프로세스 관리 로직(생성, 종료, 상태 변경, 스케줄링 등)이 구현되어 있는 파일입니다.
- **분석**: `exit()`나 `exit2(status)` 함수 내부의 실제 로직이 동작하는 공간입니다. 열려있는 파일 디스크립터를 모두 닫고, 현재 디렉토리 참조를 반환(`iput`)하며, 잠들어 있을지 모르는 부모 프로세스를 깨웁니다(`wakeup1`). 자신의 자식 프로세스들이 있다면 고아가 되지 않도록 `init` 프로세스에게 입양시키고, 최종적으로 자신의 프로세스 상태를 `ZOMBIE`로 변경한 후 스케줄러(`sched()`)를 호출하여 CPU 제어권을 다른 프로세스에게 넘깁니다. (특히 새로 추가된 `exit2`의 경우 넘겨받은 `status`를 `proc->xstate`에 저장하는 로직이 여기서 실행됩니다.)

### 7. `proc.h` (PCB 및 프로세스 자료구조 정의)
- **역할**: 프로세스를 추상화하고 제어하기 위한 핵심 자료구조인 PCB(Process Control Block) 역할의 `struct proc` 구조체와 프로세스 상태 상수 등을 정의합니다.
- **분석**: `proc.c`에서 프로세스의 상태를 관리하고 자원을 할당/해제하기 위해 필요한 기본 뼈대입니다. 기존 `exit` 메커니즘에서는 구조체의 변경이 없었으나, `exit2`와 `wait2` 메커니즘을 완성하기 위해 이 파일의 `struct proc` 구조체 내에 자식의 종료 코드를 저장할 수 있는 공간인 `int xstate;`를 새롭게 추가함으로써 커널 자료구조의 업데이트가 이루어졌습니다.

---

### 📝 흐름 요약
1. **[User]** 응용 프로그램이 `user.h`에 선언된 `exit()` 함수 호출
2. **[User -> Kernel]** `usys.S`에서 `SYS_exit` 번호를 담아 `int $T_SYSCALL` 소프트웨어 인터럽트 발생 (트랩)
3. **[Kernel]** `syscall.h`에 정의된 번호를 바탕으로 `syscall.c`의 `syscalls` 배열에서 디스패치
4. **[Kernel]** `sysproc.c`의 `sys_exit()`에서 필요한 경우 유저 공간 인자 검증 및 추출 수행
5. **[Kernel]** `proc.c`의 `exit()`로 넘어가 자원 정리 및 상태를 `ZOMBIE`로 변경하고 스케줄링
6. **[Kernel]** 이 모든 로직은 `proc.h`에 정의된 `struct proc` (PCB) 구조체를 기반으로 상태 및 데이터를 업데이트하며 진행됨
