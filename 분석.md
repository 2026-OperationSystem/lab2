# xv6 시스템 콜 분석 및 기존 exit() / wait()의 문제점

본 문서에서는 xv6 운영체제의 기존 `exit()` 및 `wait()` 시스템 콜의 동작 원리와 한계점을 분석하고, 이를 해결하기 위해 새로 구현된 `exit2()`, `wait2()` 시스템 콜과의 비교를 다룹니다.

## 1. 기존 xv6의 `exit()` 및 `wait()` 분석

### 기존 동작 방식
- `void exit(void)`: 현재 실행 중인 프로세스를 종료합니다. 열려 있는 모든 파일 디스크립터를 닫고, 현재 디렉토리의 참조를 해제한 뒤, 프로세스 상태를 `ZOMBIE`로 변경하고 잠들어있을지 모를 부모 프로세스를 깨웁니다(wakeup). 이 때 전달받는 인자(매개변수)가 없습니다.
- `int wait(void)`: 부모 프로세스가 자식 프로세스의 종료를 기다립니다. 프로세스 테이블을 순회하며 `ZOMBIE` 상태인 자식 프로세스를 찾으면, 해당 자식의 커널 스택과 페이지 테이블 등 메모리 자원을 해제하고 자식의 PID를 반환합니다. 인자가 없으며, 단순히 종료된 자식의 PID만 반환합니다.

### 기존 시스템 콜의 문제점 (한계점)
표준 UNIX/Linux(POSIX) 시스템에서는 프로세스가 종료될 때 성공(0) 또는 실패(에러 코드)를 의미하는 **종료 상태 코드(Exit Status)** 를 반환하며, 부모 프로세스는 이를 확인하여 자식 프로세스의 작업 성공 여부와 결과를 파악할 수 있습니다. 

하지만 기존 xv6의 `exit()`와 `wait()`는 다음과 같은 명확한 문제점과 한계를 가집니다.
1. **종료 상태 정보 전달 불가**: 자식 프로세스가 어떤 이유로 종료되었는지(정상 종료인지, 에러로 인한 종료인지) 부모 프로세스가 알 수 있는 방법이 전혀 없습니다. (`exit`에 인자가 없고, 커널의 `proc` 구조체에도 종료 상태를 저장할 변수가 존재하지 않음)
2. **프로세스 제어 및 협력의 제한**: 쉘(Shell) 스크립트나 부모 프로세스에서 자식 프로세스의 실행 결과에 따라 조건부 분기(예: `cmd1 && cmd2`)를 수행해야 할 때, 실행 결과를 알 수 없으므로 이러한 흐름 제어를 구현할 수 없습니다.

## 2. 새로 구현한 `exit2()` 및 `wait2()`와의 비교

이러한 문제점을 해결하기 위해 구현된 `exit2()`와 `wait2()`는 부모-자식 간에 상태 값을 전달할 수 있도록 확장되었습니다.

| 구분 | 기존 xv6 (`exit` / `wait`) | 개선된 xv6 (`exit2` / `wait2`) |
| --- | --- | --- |
| **인자 (Arguments)** | `exit(void)`, `wait(void)` | `exit2(int status)`, `wait2(int *status)` |
| **상태 저장 공간 (`proc`)** | 종료 상태를 저장할 공간 없음 | `int xstate;` 변수를 추가하여 자식의 종료 코드 저장 |
| **종료 (exit) 동작** | 상태값 전달 없이 자원을 정리하고 ZOMBIE로 변경 | 인자로 넘겨받은 `status` 값을 자신의 `proc->xstate`에 저장한 후 ZOMBIE로 변경 |
| **대기 (wait) 동작** | ZOMBIE 자식의 자원을 해제한 후 PID만 반환 | ZOMBIE 자식을 찾으면 `proc->xstate` 값을 `status` 포인터 위치에 복사하여 부모에게 전달하고 자원 해제 후 PID 반환 |

### 개선의 의의
새로 추가된 구현을 통해 부모 프로세스는 자식 프로세스가 `exit2(status)`를 통해 남긴 **종료 코드(Exit Status)를 확인**할 수 있게 되었습니다. 
이 변경점은 운영체제에서 부모-자식 프로세스 간의 가장 기초적인 정보 전달 채널을 완성한 것이며, 실제 상용 운영체제(Linux 등)와 동일한 메커니즘을 갖추게 되었습니다.

프로젝트 내에 추가된 `testwait.c`를 보면 자식 프로세스는 `exit2(37)`로 종료하고, 부모 프로세스는 `wait2(&status)`를 호출하여 그 상태값을 받아 출력(`Child exited with code 37`)하도록 되어있어, 기존의 단방향 종료 메커니즘이 가진 문제를 완벽히 해결했음을 증명합니다.
