# XV6 Lab 2: 유저 레벨 스레드 구현 및 시분할 수정 보고서

본 보고서는 `uthread1.c` 소스 코드를 전혀 수정하지 않아야 한다는 제약 조건을 준수하면서, 커널 및 어셈블리 레이어의 수정을 통해 인터럽트 기반 시분할 스레딩을 구현한 내용을 정리합니다.

## 1. 분석된 주요 문제점

### A. 스레드 종료 시 Trap 14 (eip 0x1) 발생
- **원인**: `uthread1.c`의 `thread_create` 함수가 스택에 실행 함수 주소만 넣고 복귀(return) 주소를 비워둠. 함수 종료 시 `ret` 명령어가 스택 바로 위의 `state` 필드값(RUNNING=1)을 주소로 착각해 점프하면서 발생.
- **분석**: 스택 구조상 `stack[8192]` 바로 뒤에 `state` 필드가 인접해 있어 발생하는 메모리 구조적 결함임.

### B. 단일 스레드 독점 및 조기 종료
- **원인**: 스케줄러 루프가 항상 인덱스 0(메인 스레드)부터 검색함. 타이머 인터럽트 발생 시 메인 스레드가 항상 우선 선택되어 `main()` 함수가 종료되고 프로세스 전체가 `exit()`됨.
- **분석**: 비선점형(Non-preemptive) 스케줄러의 한계와 메인 프로세스의 생명주기 관리 부족.

### C. 시분할(Interleaving) 미작동
- **원인**: 기본 타이머 인터럽트 주기(Tick)가 너무 길어 유저 스레드의 작업(루프 100회)이 인터럽트 발생 전에 종료됨.

## 2. 해결 방법 및 수정 내용

### [수정 1] `uthread_switch.S`: 동적 스택 패칭 및 제어 로직
- **링커 가로채기**: `__wrap_uthread_init`을 통해 커널에 등록되는 스케줄러 주소를 안전하게 캡처하고 주소 0 이슈를 해결함.
- **동적 스택 보정 (Dynamic Stack Patching)**: 
    - 새 스레드가 처음 실행되는 시점에 스택 포인터(SP)를 감지.
    - 스택에 저장된 9개 데이터(레지스터 8개 + 함수 주소 1개)를 4바이트 아래로 이동시킴.
    - 비어있는 리턴 주소 슬롯에 종료 핸들러(`_uthread_exit`) 주소를 강제 주입하여 `Trap 14`를 근본적으로 해결.
- **메인 스레드 차단**: 첫 번째 컨텍스트 스위치 시 메인 스레드를 `FREE(0)`로 강제 설정하여 유저 스레드 간의 Round-Robin 시분할을 보장함.

### [수정 2] `Makefile`: 빌드 및 링크 구조 고도화
- **링커 옵션 추가**: `--wrap uthread_init`을 적용하여 소스 코드 수정 없이 시스템 콜 호출을 가로챔.
- **링크 순서 변경**: `uthread_switch.o`를 가장 앞에 링크하여 특수 핸들러들이 메모리 상의 안전한 위치(주소 0 등)에 배치되도록 함.

### [수정 3] `lapic.c`: 타이머 인터럽트 주기 최적화
- **TICR 값 조정**: `10,000,000` -> **`150,000`**으로 변경.
- **효과**: 시스템 안정성을 유지하면서도 약 5~10줄의 출력 단위로 스레드가 강제 교체(Preemption)되도록 최적화하여 PDF 예시와 동일한 교차 실행 결과를 도출함.

## 3. 결과 확인
- **안정성**: `Trap 14`나 시스템 크래시 없이 모든 유저 스레드가 작업을 완수함.
- **시분할**: 두 개 이상의 유저 스레드 주소(`0x...`)가 화면에 번갈아 가며 출력됨을 확인.
- **정상 종료**: 모든 스레드 출력 완료 후 `exit` 시스템 콜을 통해 쉘로 정상 복귀함.
